# Component Generation Rules

> **Foundation Source:** `https://ui.shadcn.com/create?preset=b2H8qFFOtM`\
> **Preset ID:** `b2HRtJbF9k`\
> **Aesthetic Directive:** Avant-Garde Cyan/Mist — Intentional Minimalism\
> **Version:** 2.0 (Strict Enforcement)

---

## Overview

This document defines the state-machine workflow for generating and styling UI
components. All output must enforce strict Shadcn registry compliance, semantic
variable usage, and rectilinear geometry. No exceptions.

---

## State 1: Verification

Before writing a single line of component code, the agent must:

1. Check if the requested UI primitive exists in the local `components.json`
   manifest.
2. Cross-reference the official Shadcn UI registry
   (`https://ui.shadcn.com/docs/components`).
3. **Condition:** If the primitive exists in either source, proceed to **State
   2**.
4. **Condition:** If the primitive does **NOT** exist, proceed to **State 1a**.
5. Do not proceed to scaffolding without completing this gate.

---

## State 1a: Acquisition

If the primitive is missing from the local environment:

1. Execute `npx shadcn@latest add [component]` to acquire the primitive from the
   official registry.
2. Wait for successful installation confirmation.
3. Verify the primitive appears in `components.json` or the expected file tree
   (`src/components/ui/`).
4. If installation fails, abort generation and report the failure. Do not
   attempt to build the component from scratch.
5. Upon successful verification, proceed to **State 2**.

---

## State 2: Import & Scaffolding

1. Import the verified Shadcn primitive into the target custom component file.
2. Build the component structure using **semantic HTML5** elements (`<section>`,
   `<article>`, `<header>`, `<nav>`, `<button>`, etc.).
3. Wrap the primitive only when absolutely necessary to achieve the
   "Avant-Garde" aesthetic.
4. **ABSOLUTE RULE — NO CURVED GEOMETRY:** All elements must be rectilinear.
   Curved lines, rounded corners, and circular geometry are **FORBIDDEN**.
   - **Banned classes & properties:** `rounded`, `rounded-sm`, `rounded-md`,
     `rounded-lg`, `rounded-xl`, `rounded-2xl`, `rounded-3xl`, `rounded-full`,
     `border-radius`, `circle`, `ellipse`, `radial-gradient`.
   - **Mandatory override:** If a base Shadcn primitive ships with default
     rounding, you **must** override it explicitly with `rounded-none`.
   - All corners must remain at `0px` (sharp 90-degree angles).

---

## State 3: Variable Mapping

1. Apply **only** semantic Tailwind variables derived from the `b2HRtJbF9k`
   preset.
2. **ABSOLUTE RULE — VARIABLE TAXONOMY ONLY:** Use exclusively the CSS variables
   generated by the shadcn CLI in `src/index.css`.
3. **Banned Hardcoding (Zero Tolerance):**
   - Raw hex codes (`#FF0000`, `#0ff`, `#000000`)
   - RGB / RGBA values (`rgb(0,0,0)`, `rgba(255,0,0,0.5)`)
   - HSL / OKLCH literals (`hsl(200,50%,50%)`, `oklch(50% 0.2 200)`)
   - Inline style color values (`style={{ color: 'red' }}`)
   - Tailwind arbitrary bracket values for color (`bg-[#F59E0B]`,
     `text-[rgb(0,0,0)]`)
4. **Allowed Usage:**
   - Standard Tailwind layout utilities (`flex`, `grid`, `p-4`, `gap-2`,
     `w-full`, etc.)
   - Global semantic variables: `bg-background`, `text-foreground`,
     `border-border`, `ring-ring`
   - Primary/Secondary variables: `bg-primary`, `text-primary-foreground`,
     `bg-secondary`, `text-secondary-foreground`
   - State variables: `bg-muted`, `text-muted-foreground`, `bg-accent`,
     `text-accent-foreground`, `bg-destructive`, `text-destructive-foreground`
   - Opacity modifiers on allowed variables only: `bg-primary/90`,
     `text-muted/50`
   - Border and ring variables: `border-border`, `ring-ring`
   - Sharp geometry override: `rounded-none`

---

## State 4: Validation

Before marking any component as complete, run the following audit:

1. **Minimalism Check:** Are there unnecessary DOM elements? If yes, remove
   them.
2. **Taxonomy Check:** Does every style map exactly to the Great Forge taxonomy
   (`b2HRtJbF9k` preset variables)?
3. **Hardcode Check:** Is the component 100% free of hardcoded color values?
   Scan every class, inline style, and logic branch.
4. **Geometry Check:** Is the component 100% free of curved geometry? Audit
   every class and inline style for forbidden radius/curve properties.
5. **Contrast Check:** If utilizing custom variable combinations (e.g.,
   `bg-muted` + `text-accent`), verify WCAG AAA contrast compliance. If the
   combination fails, remap to compliant variables.

**If any check fails, the component is rejected and must be self-corrected
before final output.**

---

## Self-Correction Protocol

Any generation attempt failing the above rules **must** be automatically
rejected by the agent. The agent **must** self-correct immediately by:

1. Stripping all hardcoded color values.
2. Stripping all curved geometry classes and properties.
3. Re-mapping all styles to the defined variable taxonomy.
4. Re-auditing the entire file against this document before final output.
5. Do not deliver code to the user until all checks pass.

---

## Summary of Non-Negotiables

| Rule                                                  | Status                      |
| ----------------------------------------------------- | --------------------------- |
| Use only `b2HRtJbF9k` preset variables                | Mandatory                   |
| Use only sharp, rectilinear geometry (`rounded-none`) | Mandatory                   |
| No hardcoded colors (hex, rgb, hsl, oklch)            | Forbidden                   |
| No curved lines, rounded corners, or circles          | Forbidden                   |
| No manual CSS variable creation                       | Forbidden                   |
| Semantic HTML5 structure                              | Mandatory                   |
| WCAG AAA contrast compliance                          | Mandatory for custom combos |
